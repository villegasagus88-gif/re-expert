## 5. Cosas a EVITAR (constraints)

Estas son reglas del proyecto. Violarlas rompe trabajo previo o expone secrets.

### 🚫 Nunca hardcodear secrets en código
- Todas las keys (Anthropic, Resend, Stripe, JWT_SECRET, Supabase, Sentry, Tavily) viajan **solo** por env vars en Railway o Netlify.
- `.env.example` lista cada una con formato esperado.
- GitHub tiene Push Protection activo — bloquea commits que contengan secrets.
- Si una API key se filtra: rotar inmediatamente en el dashboard del proveedor + rotar `JWT_SECRET` (invalida todos los tokens) + force logout global.

### 🚫 No tocar APIs externas en el chat de Claude
Este es el constraint #1 del user: **el chat NO contrata ni activa APIs
externas**. Solo documentamos qué hacer y por qué. La activación va al
dashboard que maneja Agustín (owner Supabase, Railway, Stripe) o Mati
(las que tiene él: Resend, Gemini, UptimeRobot, Sentry).

### 🚫 No tocar la knowledge base sin checkpoint
- 240 archivos en `knowledge/` en 20 carpetas temáticas. Si tocás algo
  acá, los queries del chat se rompen silenciosamente.
- Si necesitás modificar contenido, hacelo via Supabase Storage UI (es
  donde queda en prod). El folder local es solo seed inicial.

### ⚠️ Project ≠ Workspace — son dos conceptos distintos
Esto confunde a primera vista, pero es importante no mezclarlos:

| Concepto | Modelo | Cardinalidad | Para qué |
|----------|--------|--------------|----------|
| **Project** | `models/project.py` | **UNO** por usuario | Tracker de obra: presupuesto, CPI, SPI, milestones, costos por rubro. Dashboard en sección "📊 Panel de Proyecto". |
| **Workspace** | `models/workspace.py` | **N** por usuario | Agrupador tipo "carpeta de chats" de ChatGPT. Memoria propia que se inyecta como contexto cuando el chat está dentro. Sección "📁 Proyectos" del sidebar. |

Si el user dice "proyecto" en una conversación, **siempre desambiguar:**
"¿el tracker de obra (Panel de Proyecto) o un workspace de chats?".
Tener cuidado con el wording en UI nuevas para no romper el modelo
mental del user.

### 🚫 No mergear sin pasar por `merge/launch-mvp-into-main` primero
- Branch policy: feature branches → `merge/launch-mvp-into-main` → `main`.
- `main` auto-deploya. Si rompés `main`, el sitio entra a producción roto.
- Patrón usado: push directo `main` solo cuando el branch ya está mergeado con
  `origin/main` (fast-forward seguro). Ver commit `dfb3088` como ejemplo.

### 🚫 No usar `git rebase -i`, `git push --force`, `git reset --hard` en main
- Reglas explícitas del usuario. El proyecto tiene un solo dueño con merge
  permissions (Agustín) — destruir historial complica la coordinación.

### 🚫 No tocar Supabase desde el chat
- Mati no tiene Owner permission en Supabase (Agustín sí). Todo lo
  relacionado con la DB (schemas nuevos, RLS, storage policies) se
  documenta para que Agustín lo aplique.
- Migrations sí: `alembic upgrade head` corre en `preDeployCommand` de
  Railway — basta con pushear a main.

### 🚫 No instalar `gh` CLI / pedirle login al user
- En la máquina actual no hay `gh` instalado y no hay GitHub token en
  env. Para abrir PRs, dar el URL al user y que lo abra desde browser.
- Push a remoto sí funciona (vía credential.helper=manager Windows).

### 🚫 No proponer "seguir mañana" cuando hay laburo pendiente
- Constraint explícito del user: "no me sugieras más seguir con las
  tareas después, empeza con la lista que me recomendaste". Si hay
  pendientes claros, ejecutar; no preguntar permiso entre pasos.

## 6. Cómo hablamos

### Idioma y estilo
- Español rioplatense, voseo. Tono directo, sin formalidad innecesaria.
- "Tech Lead" — el user es dev avanzado, no explicar conceptos básicos.
- Cuando hay decisión de arquitectura que tiene trade-offs: **proponer la opción + razón breve**, no pedir que el user elija sin contexto.
- Sin emojis en código/docs salvo que el user pida explícitamente.

### Plan-first, ejecutar después
- Para cualquier task no-trivial: dar el plan en 1-3 bullets ANTES de tocar archivos. Después ejecutar todo el plan en una pasada.
- No hace falta confirmar entre pasos del mismo plan. Solo confirmar si aparece algo fuera de scope.

### Verificación antes de claim de "listo"
- Si decís "ya está desplegado" → mostrar HTTP code de prod.
- Si decís "tests pasan" → mostrar output del runner.
- Si decís "syntax OK" → mostrar el output del linter/checker.

import re, subprocess, tempfile, os

html = re.sub(r'<!--.*?-->', '', html, flags=re.DOTALL)
scripts = re.findall(r'<script(?![^>]*\bsrc=)[^>]*>(.*?)</script>', html, re.DOTALL)
